This specification outlines a modular framework for the Fairness Evaluator, an advanced AI model designed to analyze agreements for extraordinary or unenforceable claims based on legal principles. It aims to enhance situational awareness for legal entities, particularly natural persons, by supporting informed decision-making, liability considerations, and financial structuring. The framework incorporates RDF for semantic precision and supports various output formats suitable for LLM agents.

This is an unofficial draft specification. It is intended for discussion and refinement within the Web Civics community and beyond.

Introduction

The Fairness Evaluator Framework addresses the need for AI-assisted evaluation of legal agreements to identify unenforceable clauses due to fundamental legal principles such as illegality, unconscionability, or violations of public policy. By providing a structured approach, this framework enables software agents (e.g., LLMs) to consider complex vectors including legal, ethical, economic, social, resolution, and temporal implications.

Building on prior framing documents and data structures, this specification defines a method to support 'modules'—self-contained use-case definitions—that can be processed alongside circumstantial contexts. RDF support ensures namespace specificity, allowing for unambiguous semantic linking.

This refined definition emphasizes modularity for extensibility, semantic web integration via RDF, and agent-friendly outputs. Modules act as knowledge graphs, enabling LLMs to reason over linked data for nuanced evaluations, incorporating research on digital slavery and related implications for modern agreements.

Framework Overview

The framework consists of core components:

  • Core Evaluator Engine: An AI model (e.g., LLM-based) that processes inputs against defined principles and vectors.
  • Modules: Pluggable use-case definitions, each encapsulating specific contexts, principles, and evaluation logic.
  • Data Structures: JSON/RDF schemas for principles, vectors, and outputs.
  • Integration Layer: Supports RDF namespaces, input parsing, and output in formats like JSON-LD, Markdown-LD*, or Turtle.

The evaluator operates in a pipeline: Input agreement text → Parse clauses → Match against modules/principles → Evaluate vectors → Generate report with implications.

Modules and Use-Cases

Modules are defined as RDF resources or JSON-LD documents, each representing a use-case (e.g., 'prohibition_of_slavery_in_contracts'). A module includes:

Namespace
A unique RDF namespace (e.g., http://example.org/ns/slavery#).
Context Documents
References to legal bases, examples, and variations.
Evaluation Logic
Prompt templates or rules for LLM processing.
Circumstantial Integration
Parameters for injecting real-time context (e.g., jurisdiction, party details).

Agents load modules dynamically, combining them for comprehensive evaluations. For instance, a contract analysis might invoke 'digital_slavery' and 'unconscionability' modules, processing implications from research on data alienation and behavioral control.

RDF Support and Semantic Precision

To ensure specificity, all key terms use RDF namespaces. For example:

@prefix fe: <http://example.org/fairness-evaluator#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .

fe:Principle rdf:type rdf:Class ;
    rdfs:label "Legal Principle" .

Outputs can be serialized in RDF/JSON (JSON-LD), allowing linked data integration. Markdown-LD* extends Markdown with embedded JSON-LD for human-readable yet machine-parseable reports, facilitating LLM ingestion.

Data Structures

The primary data structure is a JSON schema with RDF mappings:

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "namespace": { "type": "string" },
    "description": { "type": "string" },
    "legal_basis": { "type": "array", "items": { "type": "object" } },
    // ... (as per previous examples)
  }
}

For LLM compatibility, outputs default to JSON-LD, with options for Turtle or Markdown-LD* to balance readability and semantics.

Implementation Guidelines

Implementations should:

  1. Support loading RDF datasets for modules.
  2. Use SPARQL for querying principles.
  3. Generate reports balancing human readability and machine parsability.
  4. Ensure bias mitigation in evaluations.
  5. Incorporate research implications, such as digital slavery vectors, for holistic risk assessment.
